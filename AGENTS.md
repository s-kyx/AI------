# AGENTS.md

## 项目定位

**先做一点**是一个本地优先的 AI 焦虑减负工具：用户把脑中混乱的事情一次性写下，应用将其整理为按日期和时间排列、能立刻开始的待办。它的首要目标是降低心理负荷与启动阻力，而不是提高任务吞吐量，更不是一个 GTD 或项目管理系统。

目标用户是被工作、学习和生活事项压得焦虑的年轻职场人和学生；他们不愿学习复杂方法，也不应该被要求填写优先级、标签、工时或截止日期等表单。

## 当前技术与边界

- 单页前端：React 19 + Vite 7，入口为 `src/main.jsx`，样式为 `src/styles.css`。
- 当前为纯前端、本地优先产品；**不要默认增加后端、登录、云同步、数据库或埋点**。这类改动需要用户明确授权。
- 用户自行在右上角设置里填写模型 API Key；配置及结果存在浏览器 `localStorage`。不要在界面、日志、错误信息、导出内容或代码中泄露 API Key。
- 模型请求从浏览器直接发向第三方模型服务。修改请求格式、模型配置或提示词前，要保留现有模型的兼容性，并以真实 API 文档为准。
- 当前支持 DeepSeek、豆包与 GLM；默认模型是 `deepseek-v4-pro`。不要无故更换默认模型、Base URL 或推理设置。

## 核心体验（优先级最高）

1. 用户只需看到一个大的自由输入框，写下烦心事。
2. AI 返回一句短而安定的鼓励，加一份按日期/时间升序的清晰待办。
3. 用户可以勾选完成、继续补充、导出 Word/PDF。

界面应当白色、安静、克制、有轻微情绪安抚感。保留“大输入框 + 小设置齿轮 + 单一主按钮 + 结果清单”的低学习成本结构。避免仪表盘、KPI、优先级矩阵、复杂筛选、卡片堆砌、强刺激动画、催促式文案和“效率焦虑”。

## 心理健康安全原则

- 产品定位为情绪支持与行动澄清，不做诊断、治疗承诺、风险评估结论或替代专业帮助。
- 用不评判、不过度乐观、不给用户贴标签的中文；先帮助稳定与减负，再提出一小步行动。
- 如果输入出现自伤、伤害他人、明确绝望或即时危险信号，**不得继续生成普通待办清单**。应进入简短、明确的危机支持界面：鼓励立即联系身边可信任的人、当地紧急服务或专业危机支持，并提供用户所在地已核实的求助信息。
- 任何危机识别、求助文案、心理练习或临床相关能力上线前，都需要合格心理健康专业人士参与审阅，并进行安全测试；不要只依赖模型提示词。
- 不把情绪困扰、完成率或文本内容用于营销、画像或公开展示。若未来引入存储或分析，先完成明确同意、最小化采集、可删除和安全评审。

## 重要数据约定

- localStorage keys：`anxiety-planner.input.v1`、`anxiety-planner.result.v1`、`anxiety-planner.ai.v1`。
- AI 结果需保持 JSON 兼容：`summary`、`schedule`、`later`。每条 `schedule` 至少兼容 `date`、`time`、`title`、`estimated_minutes`、`buffer_minutes`、`completed`。
- `summary` 是不超过 18 个中文字的安定鼓励；用户主要看到的是可勾选、动词开头、具体可执行的 `title`。
- 排程要考虑现实缓冲、任务间空档和当天精力余量，不能把一天排满，也不能用无意义的小任务凑数量。
- 修改结果结构时必须兼容旧 localStorage 数据，并保留导出、勾选完成与重新分析流程。

## 代码组织与改动方式

- `PRODUCT.md` 是产品语气、目标用户和反模式的事实来源；触及产品行为或文案时先阅读它。
- `src/main.jsx` 当前同时承担状态、模型调用、导出和界面渲染。小修可就地完成；新增一块独立功能时，优先拆到 `src/components/`、`src/lib/` 或 `src/hooks/`，不要继续把大段逻辑堆进入口文件。
- `src/styles.css` 使用 OKLCH、慢速背景流动和 `prefers-reduced-motion` 降级。新增动效必须可关闭、不能妨碍输入或阅读。
- 保持键盘可用、清晰焦点态、可读对比度和移动端 320px 起的可用性。
- 工作树可能含用户未提交的 `口播文案.docx` 改动；除非用户明确要求，不要触碰它或回退任何现有改动。

## 验证与交付

- 本项目默认只做代码级验证：运行 `npm run build`，必要时做静态搜索和逻辑检查。
- 用户明确不需要浏览器自动化、首屏抓取或截图；除非用户在该次请求中主动要求，不要做这些检查。
- 没有现成测试框架。若添加纯逻辑模块，可在不改变产品范围的前提下补最小可运行测试。
- 在交付时说明：改了什么、运行了什么验证、尚未验证的外部行为（尤其是第三方模型 API）。

## 变更决策顺序

先保护安全与隐私，再保护低压力体验，再保护数据兼容与简单性，最后才考虑新功能、视觉装饰和“效率”指标。任何会让产品从“帮人先做一点”滑向“逼人做更多”的设计，默认不做。
